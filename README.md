# wm-skills

Skills for AI agents.

## Skills

### `system-overview`
快速理解复杂仓库或子系统，建立全局心智模型。

适用场景：
- 接手一个新模块，想先看清整体结构
- 改代码前，需要先摸清入口、分层和关键对象
- 想知道系统的主流程、线程模型和关键依赖

核心能力：
- 识别仓库或子系统边界
- 提炼启动点、初始化点、注册点和主循环
- 梳理模块分层、核心对象、线程模型与异步链
- 给出风险点和新人阅读顺序

### `trace-feature`
沿着真实调用链追踪一个功能从入口到可观察结果的执行路径。

适用场景：
- 想知道某个功能是怎么跑通的
- 已知一个函数或类，需要继续追上下游
- 改动前做影响面分析，确认会改到哪里

核心能力：
- 找到功能入口，包括事件、回调、消息或脚本入口
- 跳过低价值转发层，只保留关键调用链
- 追到具体副作用，如状态更新、资源提交、网络发送或文件读写
- 标出线程切换、异步边界、前置条件和高风险点

### `explain-symbol`
准确解释一个 symbol 在系统中的真实职责，而不是只看名字猜用途。

适用场景：
- 不清楚某个类、函数、宏、枚举、全局对象是干什么的
- 想区分两个名字相近但职责不同的概念
- 需要快速理解一个 symbol 的上下游和生命周期

核心能力：
- 判断 symbol 类型，如 class、function、macro、enum、global
- 说明它所处分层、所属模块和最接近的核心对象
- 梳理创建者、持有者、调用者、依赖项和影响范围
- 分析生命周期、线程归属、输入输出、副作用和常见误解

### `capture-web-article`
把单篇网页文章、博客或社交平台帖子归档为本地 Markdown 包，并下载正文内图片。

适用场景：
- 需要把一个网页内容保存到本地长期留档
- 需要保留渲染后的正文内容，而不是抓原始 HTML
- 页面需要登录后才能看到正文，仍然希望完整保存

核心能力：
- 要求用户提供 URL 和输出目录，并在本地创建带时间戳的归档目录
- 通过浏览器渲染后的 DOM 提取正文，而不是直接抓取原始页面
- 过滤导航、侧边栏、评论区等噪音内容，按阅读顺序保留图片
- 生成 Markdown 正文、下载图片并改写为本地引用

注意事项：
- 这个 skill 只在用户显式调用 `$capture-web-article` 时使用
- 执行前必须提供 `url` 和 `output_dir`

### `orchestrate-change`
OpenSpec 多 Agent 工作流的统一入口，负责编排规划、实现、审查和验证的完整流程。

适用场景：
- 启动一个新的 OpenSpec 变更流程
- 恢复一个已有的变更流程继续推进
- 需要在规划、实现、审查之间自动流转

核心能力：
- 解析 change_id，定位工作流状态文件
- 判断当前阶段，路由到对应的专项 skill
- 执行工作流门控规则，阻止未审批阶段的越权推进
- 在实现偏离规划时自动重启规划阶段
- 同一问题反复出现 3 轮后标记为 stalled 并暂停推进

### `author-plan`
创建和修订 OpenSpec 规划产物（proposal、design、tasks、specs）。

适用场景：
- 新变更的初始规划，通过 `/opsx:propose` 生成规划产物
- 审查反馈后的修订轮次，直接修改已有产物
- 规划方向发生重大变化，需要重新生成

核心能力：
- 初始创建必须通过 `/opsx:propose` 生成
- 修订轮次优先直接编辑已有产物，保持连续性
- 仅在范围重大变更或任务分解根本有误时才重新调用 `/opsx:propose`
- 确保每个任务都有独立可验证的验收路径

### `review-plan`
审查 OpenSpec 规划产物的完整性、架构质量和任务级可测试性。

适用场景：
- 规划产物创建或更新后，需要审查是否可以进入实现阶段
- 需要检验规划的跨产物一致性和任务分解质量

核心能力：
- 评估范围清晰度、规格可测试性、设计充分性和任务分解质量
- 逐个检查每个任务的独立可验证性和端到端验收路径
- 对范围不清、规格模糊、任务无法独立验证等问题强制拒绝
- 输出 approve 或 request_changes，附带阻塞问题列表

### `implement-task`
实现一个已批准的任务，并在交付审查前完成任务级验证。

适用场景：
- 规划审查通过后，按顺序实现单个任务
- 必须通过 `/opsx:apply` 在新子 Agent 中调用

核心能力：
- 读取任务要求和关联的 spec、design 内容
- 实现任务并添加或更新所需测试
- 运行构建、lint、单测、集成测试和任务级端到端验证
- 验证不通过则停留在实现阶段修复，直到全部通过
- 明确报告发现的 spec 或 design 缺口

### `review-code`
审查单个已批准任务的实现，检查正确性、规格对齐、设计对齐和测试充分性。

适用场景：
- 任务实现完成并通过验证后，进入代码审查
- 需要判断实现是否安全可接受

核心能力：
- 评估任务正确性、规格符合度、设计对齐度、代码质量和回归风险
- 检查测试覆盖和任务级端到端验证的真实性
- 对未满足任务要求、偏离规格、缺少测试等问题强制拒绝
- 区分阻塞问题和非阻塞质量建议，输出 approve 或 request_changes
